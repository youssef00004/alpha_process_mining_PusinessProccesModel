=== 1. INTRODUCTION ===

Process mining is a data-driven discipline that bridges business process management and
data science. It extracts knowledge about how business processes actually execute by
analysing the digital traces that processes leave behind in information systems — known
as event logs. Each record in an event log captures which activity was performed, when it
was performed, which case (process instance) it belongs to, and which resource carried
it out. By systematically analysing these records, process mining can discover the real
process model, check whether execution conforms to a prescribed model, and identify
where performance bottlenecks or deviations occur. Unlike traditional process modelling,
which relies on interviews and expert knowledge, process mining produces models that are
objectively grounded in observed behaviour.

This assignment implements the Alpha Algorithm, the foundational process discovery
algorithm introduced by van der Aalst, Weijters, and Maruster (2004). The algorithm
takes an event log as input and automatically discovers a Petri net that describes the
causal structure of the process — which activities must precede others, which can execute
in parallel, and which are mutually exclusive. The implementation covers all eight formal
steps of the algorithm, from extracting the activity set through constructing the final
Petri net. A full automation pipeline accompanies the core algorithm: it reads any
conforming Excel event log, applies the Alpha Algorithm, generates an Activity-on-Arc
(AoA) PERT chart derived entirely from the discovered Petri net, and writes a four-sheet
Excel report — all without modifying source code.


=== 2. TOOL SELECTION & JUSTIFICATION ===

Python 3 was selected as the implementation language for this assignment. The decision
was made by evaluating the specific technical requirements of each deliverable against
the capabilities of the available tools.

The Alpha Algorithm requires set-theoretic operations: powerset enumeration, causal pair
identification, maximality testing, and set intersection and subset checks. Python's
native frozenset type and itertools.combinations function map directly onto the
mathematical definitions in van der Aalst (2004), making the implementation both concise
and verifiable against the formal specification. The footprint matrix is a dictionary of
dictionaries with O(n^2) pair iteration, which reads naturally in Python and mirrors the
formal notation. Graph construction and layout for the PERT chart are handled by
NetworkX (directed graph representation and topological sort) and Matplotlib (publication-
quality rendering with precise arc and label control). Excel reading and writing are
handled by pandas and openpyxl, which support multi-sheet workbooks with auto-formatted
columns in a few lines of code. A single run_pipeline() call in automation.py processes
any conforming event log end-to-end without modifying source code, demonstrating the
reusability that object-oriented scripting enables.

Comparison with alternatives:

                | itertools, dicts  | Matplotlib        | openpyxl         | any log via CLI

                | no set types      | graph engine      | hard to script   | no graph output

                | (dplyr, sets)     | but less control  |                  | weaker PM ecosystem

Python is the correct choice because it offers the highest fidelity between the formal
mathematical algorithm and the code, the richest process mining ecosystem, and the

pipeline. Python 3.8+ with the packages listed in requirements.txt reproduces all results
on any machine with one pip install command.

=== 4. DATA PREPROCESSING ===

Before the Alpha Algorithm is applied, the raw event log is loaded and cleaned through
a series of preprocessing steps implemented in automation.py.

Column validation: The loader checks that the four required columns — Case ID, Activity,
Timestamp, and Resource — are present in the Excel file. Column names are normalised
(stripped of leading and trailing whitespace, matched case-insensitively) so that minor
formatting inconsistencies in the source file do not cause failures. An Event ID column
is optional; if absent, sequential identifiers (E1, E2, ...) are generated automatically.

Type coercion and cleaning: Case ID, Activity, and Resource values are cast to strings
and stripped of whitespace. Timestamps are parsed with pandas.to_datetime, which handles
ISO 8601 format (YYYY-MM-DD HH:MM and YYYY-MM-DD HH:MM:SS). Fully blank rows are
dropped before processing. Rows where Case ID or Activity is empty after stripping are
also removed, and a warning is printed for each row dropped.

Sorting: The cleaned dataframe is sorted first by Case ID and then by Timestamp in
ascending order. This guarantees that activity traces are reconstructed in chronological
order when the dataframe is grouped by case. The sort is stable, preserving the original
relative order of events that share the same timestamp within a case.

Cross-case boundary prevention: Traces are extracted by grouping the sorted dataframe
by Case ID. The directly-follows relation is built strictly within each group — the last
activity of one case is never treated as directly preceding the first activity of the
next case. The implementation verifies this explicitly and reports that no cross-case
pairs were introduced.

Missing value handling: Any row where Case ID or Activity is blank after coercion is
discarded with a printed warning. If the resulting dataframe is empty, the pipeline
raises a descriptive ValueError rather than silently producing an empty result.


=== 5. ALPHA ALGORITHM — STEP BY STEP ===

The Alpha Algorithm (van der Aalst et al., 2004) operates in eight sequential steps.
Each step is described below with the actual results obtained from the four-case event log.

--- Step 1: Extract all activities (T_L) ---

Every distinct activity name that appears at least once in the event log is collected
into the set T_L. For this log, T_L contains 12 activities:

  T_L = { Archive order, Check materials availability, Check stock availability,
           Confirm order, Emit invoice, Get shipping address, Manufacture product,
           Obtain raw materials, Receive payment, Request raw materials,
           Retrieve product from warehouse, Ship product }

--- Step 2: Extract start activities (T_I) ---

The start activity set T_I contains every activity that appears as the first event in
at least one case. In all four cases the process begins with the same activity:

  T_I = { Check stock availability }

--- Step 3: Extract end activities (T_O) ---

The end activity set T_O contains every activity that appears as the last event in at
least one case. In all four cases the process ends with the same activity:

  T_O = { Archive order }

--- Step 4: Build the footprint matrix ---

For every ordered pair of activities (A, B), the directly-follows relation records
whether A is ever immediately followed by B within the same case. The footprint matrix
classifies each pair into one of four relations:

  A  →  B  : A directly precedes B in at least one case, but B never directly precedes A.
  A  ←  B  : B directly precedes A in at least one case, but A never directly precedes B.
  A  || B  : Both A→B and B→A occur across the log — the two activities are concurrent.
  A  #  B  : Neither A→B nor B→A ever occurs — the two activities are causally unrelated.

The extraction process verified that no directly-follows pair spans a case boundary.
Eighteen unique directly-follows pairs were found across the four cases.

Two concurrent (||) pairs were identified. These are the only positions in the 12×12
matrix where the || symbol appears:

  Emit invoice     ||  Get shipping address
  Receive payment  ||  Ship product

These pairs have a symmetric directly-follows relationship: in some cases Emit invoice
is followed immediately by Get shipping address, and in other cases Get shipping address
is followed immediately by Emit invoice. The same symmetric pattern holds for Receive
payment and Ship product. This symmetry is the formal signature of concurrency in the
Alpha Algorithm.

--- Step 5: Identify candidate places ---

A candidate place is a pair (A_set, B_set) of non-empty activity subsets that satisfies
three conditions simultaneously:
  (i)   Every activity in A_set has a → relation to every activity in B_set.
  (ii)  No two activities within A_set are causally related to each other (all # within A).
  (iii) No two activities within B_set are causally related to each other (all # within B).

Systematic enumeration of all valid (A_set, B_set) combinations satisfying these
conditions yielded 18 candidate places before the maximality pruning step.

--- Step 6: Remove non-maximal places ---

A place (A, B) is non-maximal if there exists another place (A', B') such that A ⊆ A'
and B ⊆ B'. Non-maximal places are redundant because the larger place already captures
all the causal information of the smaller one. Two additional consolidation passes are
applied:

  Pass 6b — Same output set, independent inputs: if two places share the same B_set
  and their A_sets have no causal relation between them, the two places are merged into
  one place with the union of their A_sets.

  Pass 6c — Same input set, independent outputs: if two places share the same A_set
  and their B_sets have no causal relation between them, the two places are merged into
  one place with the union of their B_sets.

After pruning, 9 maximal places remain:

  p1:  { Check materials availability }                            →  { Manufacture product, Request raw materials }
  p2:  { Check stock availability }                                →  { Check materials availability, Retrieve product from warehouse }
  p3:  { Confirm order }                                           →  { Emit invoice, Get shipping address }
  p4:  { Emit invoice }                                            →  { Receive payment }
  p5:  { Get shipping address }                                    →  { Ship product }
  p6:  { Receive payment, Ship product }                           →  { Archive order }
  p7:  { Request raw materials }                                   →  { Obtain raw materials }
  p8:  { Check materials availability, Obtain raw materials }      →  { Manufacture product }
  p9:  { Manufacture product, Retrieve product from warehouse }    →  { Confirm order }

--- Step 7: Construct the Petri net ---

Two special places are added to complete the Petri net:
  p_start  (source place): enables Check stock availability at the beginning of every case.
  p_end    (sink place):   receives a token when Archive order completes.

The complete Petri net consists of the 9 discovered places plus p_start and p_end,
with arcs connecting each activity to its input and output places as defined by the
(A_set, B_set) pairs.

--- Step 8: Full result summary ---

  Activities (T_L):  12 distinct activities
  Start (T_I):       Check stock availability
  End (T_O):         Archive order
  Discovered places: 9 (plus p_start and p_end)
  Directly-follows:  18 pairs extracted from 4 cases (36 events)

The full step-by-step trace is saved to alpha_algorithm_output_Input.txt.


=== 6. PROCESS UNDERSTANDING & INTERPRETATION ===

The discovered Petri net reveals an order fulfilment process with two major decision
branches at the start, followed by a common convergence point, and two concurrent
sub-processes at the end.

The initial branching point is Check stock availability (p_start → p2). Place p2 enables
two activities simultaneously: Retrieve product from warehouse and Check materials
availability. In practice exactly one path is taken per case — the Alpha Algorithm
models this as an AND-split because it cannot distinguish XOR from AND based solely on
the directly-follows relation. This is a known limitation of the algorithm.

Four distinct execution paths are present in the log:

  Path 1 — Stock available, warehouse fulfilment (Cases 1 and 4):
    Check stock availability → Retrieve product from warehouse → Confirm order → ...
    This is the fastest path. The product is already in stock, so no procurement or
    manufacturing is needed. The warehouse operative retrieves it directly.

  Path 2 — Manufacturing required, no procurement (Case 3):
    Check stock availability → Check materials availability → Manufacture product
    → Confirm order → ...
    Materials are already available, so the product is manufactured directly without
    requesting raw materials. This is a medium-length path.

  Path 3 — Full procurement and manufacturing (Case 2):
    Check stock availability → Check materials availability → Request raw materials
    → Obtain raw materials → Manufacture product → Confirm order → ...
    Raw materials must first be procured before manufacturing can begin. This is the
    longest path and involves the most resources.

After Confirm order, all paths merge and the process continues identically for every
case. Place p3 enables two concurrent sub-processes:

  Concurrent pair 1 — Invoice and payment:
    Confirm order → Emit invoice → Receive payment
    The invoice is generated and payment is collected. This sub-process is handled by
    SYS2 and financial staff.

  Concurrent pair 2 — Shipping address and dispatch:
    Confirm order → Get shipping address → Ship product
    The delivery address is retrieved and the product is physically dispatched. This
    sub-process is handled by SYS2 and warehouse/shipping staff.

The || relation between Emit invoice and Get shipping address, and between Receive
payment and Ship product, confirms that these two sub-processes execute in parallel.
In Cases 1 and 4 the address is retrieved before the invoice is issued; in Cases 2 and 3
the invoice is issued first. In Cases 1 and 4 payment is collected before shipping; in
Cases 2 and 3 the product is shipped before payment is collected. Neither ordering is a
prerequisite for the other, which the footprint matrix correctly encodes as ||.

Place p6 acts as an AND-join: Archive order can only fire when both Receive payment and
Ship product have completed, regardless of which finished first. This correctly models
the business constraint that an order cannot be closed until both the financial and
physical fulfilment obligations are met.

No loops were detected. No activity appears as both a direct predecessor and a direct
successor of another activity in a cyclic way. The discovered Petri net is acyclic,
consistent with a linear order fulfilment process where each activity executes at most
once per case. No anomalies were found: all 36 events across all 4 cases are consistent
with the discovered model, and a manual token replay of each case confirms that the
model accepts every observed trace without deadlock or remaining tokens.


=== 7. PERT CHART — EVENT NETWORK DIAGRAM ===

The PERT chart generated for this assignment uses Activity-on-Arc (AoA) notation, which
is the conventional representation for event network diagrams. In AoA notation, nodes
represent events (states of the process — points in time at which certain activities
have completed and others become enabled), and directed arcs represent activities (the
work that transitions the process from one event to the next). Each arc is labelled with
the activity name and drawn in a colour that reflects its structural role in the process.

The chart is derived entirely from the Alpha Algorithm output stored in the alpha_result
dictionary. No activity names, node positions, node counts, or arc colours are
hardcoded. The build_pert_graph_from_alpha() function in pert_chart.py translates the
9 discovered Petri net places into graph nodes and arcs through the following logic:

  - Each alpha place (A_set, B_set) becomes an intermediate event node.
  - p_start becomes the single green start node (labelled "0 — Start").
  - p_end_Archive_order becomes the single red end node.
  - For each activity, an arc is drawn from its input place node to its output place node,
    labelled with the activity name.

Two structural patterns require special handling to preserve the AoA constraint that
each activity label appears exactly once on the chart:

  Fan-out (one source place, one activity, multiple destination places):
    A grey split-event node is inserted. One labelled arc goes from the source to the
    split node; unlabelled dummy arcs (dashed, grey) go from the split node to each
    destination. This occurs for Check materials availability, which fans out to both
    p7 (Request raw materials path) and p8 (direct manufacture path).

  Fan-in (multiple source places, one activity, one destination place):
    A grey merge-event node is inserted. Unlabelled dummy arcs arrive at the merge node
    from each source; one labelled arc departs from the merge node to the destination.
    This occurs for Manufacture product, which can be enabled either via p8 alone (direct
    manufacturing) or via the procurement path through p8 with Obtain raw materials.

Dummy arcs represent zero-duration dependency constraints. They carry no activity label
and are drawn as dashed grey lines to distinguish them visually from real activity arcs.
Their sole purpose is to enforce the correct ordering of events on the chart without
duplicating activity labels.

Parallel paths are shown by arcs departing from the same node at different vertical
positions and converging again at a later node. The two concurrent pairs — Emit invoice
with Get shipping address, and Receive payment with Ship product — appear as two
separate arcs leaving the Confirm order event node (p3) and two separate arcs arriving
at the Archive order event node (p6 / p_end), making the concurrency immediately
visible in the diagram.

Node numbering is assigned by topological order of the graph. Every node receives a
unique integer label that increases monotonically from start (0) to end. This numbering
scheme is consistent with standard PERT chart conventions and is computed algorithmically
from the graph structure, not hardcoded.

=== 8. AUTOMATION ===

The automation pipeline is implemented in automation.py. It accepts any Excel event log
file that conforms to the required column specification and produces all four output
artefacts — the algorithm trace text file, the PNG and PDF PERT charts, and the four-
sheet Excel report — without any modification to source code.

Reading and validating the event log (Step 1):
The load_event_log() function accepts the path to any .xlsx file. It reads the file with
pandas.read_excel(), normalises column names to handle case and whitespace variations,
validates that all required columns are present, coerces types, drops blank rows, and
sorts by Case ID then Timestamp. Descriptive error messages are raised for missing
columns, unreadable files, and empty logs. Event IDs are auto-generated if not present.

Running the Alpha Algorithm (Step 2):
The df_to_traces() function converts the cleaned dataframe to a dictionary mapping each
Case ID to its ordered list of activities. This dictionary is passed to
run_alpha_algorithm() from alpha_algorithm.py, which executes all eight steps and
returns a structured result dictionary containing the activities, start and end
activities, the footprint matrix, the discovered places, and the directly-follows pairs.
The full step-by-step trace is simultaneously written to the output text file.

Generating the PERT chart (Step 3):
The alpha_result dictionary is passed directly to draw_pert_chart() in pert_chart.py.
The function builds the complete AoA graph, computes a topological layout with dynamic
vertical scaling, draws all nodes and arcs, and saves the chart as both a 300 DPI PNG
and a PDF. No parameters other than the alpha_result and output file paths are needed.

Writing the Excel report (Step 4):
The write_excel_report() function creates a four-sheet workbook using pandas ExcelWriter
with the openpyxl engine. Column widths are auto-fitted based on content length. The
footprint matrix is rendered with the →, ←, ||, and # symbols in a square N×N layout.

Using automation.py with a new event log:
From the command line:
  python automation.py your_log.xlsx

From Python:
  from automation import run_pipeline
  run_pipeline("your_log.xlsx", out_dir="output/")

The only requirement is that the Excel file contains columns named Case ID, Activity,
Timestamp, and Resource. Event ID is optional and will be auto-generated. Column names
are matched case-insensitively. The pipeline has been tested on the provided 4-case log
(Input.xlsx / sample_input.xlsx) and the 100-case log
(Order_Process_Log_100_Cases.xlsx), both of which produce valid output with different
discovered process structures, confirming that the pipeline is genuinely generic.


=== 9. EXCEL REPORT DESCRIPTION ===

The process mining report is saved as a four-sheet Excel workbook. Each sheet is
described below.

Sheet 1 — Event Log:
Contains the cleaned and sorted event log as a flat table. Columns are: Case ID,
Event ID, Timestamp, Activity, and Resource. Rows are sorted by Case ID ascending and
then by Timestamp ascending within each case. Timestamps are formatted as
YYYY-MM-DD HH:MM for readability. This sheet provides the auditable input data that
the Alpha Algorithm was applied to.

Sheet 2 — Footprint Matrix:
Contains the N×N causal relation matrix where N is the number of distinct activities.
Activities appear as both row labels and column headers in alphabetical order. Each cell
contains one of four symbols: → (A directly precedes B only), ← (B directly precedes A
only), || (both directions occur — concurrent), or # (no direct precedence in either
direction). The two concurrent pairs — Emit invoice || Get shipping address and
Receive payment || Ship product — are visible as || entries in the matrix. This sheet
is the formal basis for Steps 5 and 6 of the Alpha Algorithm.

Sheet 3 — Alpha Results:
Contains a two-column summary of the Alpha Algorithm output. The first column is a type
label and the second column is the corresponding value. The sheet lists the start
activities (T_I), end activities (T_O), all activities (T_L), and each of the 9
discovered places in the form "Input: {A_set} → Output: {B_set}", followed by the
source and sink places (p_start and p_end).

Sheet 4 — Process Summary:
Contains key statistics about the event log and the discovered process, including: number
of cases, number of events, number of unique activities, number of unique resources,
average case duration in hours, minimum case duration, maximum case duration, the date
range start, and the date range end. These metrics give a concise quantitative overview
of the process without requiring the reader to inspect the raw event log.

=== 10. USABILITY & REUSABILITY ===

The system is designed to work with any event log that conforms to the column
specification, without modifying source code. The following describes how to use it.

Required column names:
  Case ID     — String or integer identifier for the process instance.
  Activity    — Name of the activity performed (any non-empty string).
  Timestamp   — Date and time in ISO format (YYYY-MM-DD HH:MM or YYYY-MM-DD HH:MM:SS).
  Resource    — Name of the person or system that performed the activity.
  Event ID    — Optional. Auto-generated as E1, E2, ... if absent.

How to run with the provided log:
  python main.py
  (interactive menu — select the desired Excel file by number or name)

  Or non-interactively:
  python main.py Input.xlsx
  python main.py Order_Process_Log_100_Cases.xlsx

How to run with a new log:
  python automation.py your_new_log.xlsx

The pipeline auto-detects all Excel files in the project folder and presents them as
numbered menu options when main.py is run interactively. Output files are named after
the input file stem (e.g., pert_chart_YourLog.png) so multiple logs can be processed
without overwriting each other's outputs.

Validation on the 100-case log:
The 100-case log (Order_Process_Log_100_Cases.xlsx) contains 100 cases, 503 events,
7 distinct activities, and 7 resources. Running the full pipeline on this log produces

Error handling is built into every step of the pipeline. Informative messages are
